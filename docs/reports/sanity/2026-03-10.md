# Deep Audit - Browser-Based Datacenter Training Simulator

Date: 2026-03-10
Scope: Independent audit of the browser-based datacenter training simulator for NCP-AII exam prep

## Critical

**[SEVERITY: Critical] [AREA: Logic/Learning]**  
**File(s):** `src/components/ExamWorkspace.tsx:154`, `src/store/simulationStore.ts:620`  
**Finding:** Full-practice exam results are calculated and then discarded.  
**Detail:** `ExamWorkspace` computes a real `breakdown` at submit time, but `endExam()` ignores it and writes a placeholder all-zero breakdown instead. The placeholder is obviously unfinished and internally inconsistent: `totalPoints` is `35`, while the hardcoded domain totals add up to `33`. The results UI reads `activeExam.breakdown`, so users are shown fabricated scores.  
**Recommendation:** Replace `endExam()` with `endExam(breakdown)` or a dedicated `finishExam(breakdown)` action, persist the real breakdown, and add an integration test for submit and timeout flows. Effort: M

**[SEVERITY: Critical] [AREA: Learning/UX]**  
**File(s):** `src/components/ExamGauntlet.tsx:105`, `src/components/ExamGauntlet.tsx:166`, `src/components/ExamGauntlet.tsx:259`, `src/components/ExamGauntlet.tsx:545`  
**Finding:** The gauntlet is not an exam; it is self-certification with a progress bar.  
**Detail:** Every scenario is force-labeled as `tier: 2`, the score is just `completedCount / scenarios.length`, and the user can click `Mark Complete` without doing any scenario work. The component also claims to select from "unlocked content", but there is no unlock-state filtering in `buildAvailableScenarios()`. On top of that, `handleMarkComplete()` computes the next scenario from stale pre-update state.  
**Recommendation:** Remove manual completion, require completion tokens from the actual scenario validation pipeline, and filter by real unlock state before selection. If that is not built yet, do not market this as an exam. Effort: L

## High

**[SEVERITY: High] [AREA: Logic]**  
**File(s):** `src/components/Terminal.tsx:212`, `src/utils/scenarioLoader.ts:339`  
**Finding:** Scenario sandbox binding in the terminal can stay pointed at global state after a scenario starts.  
**Detail:** The effect that installs `scenarioContext` depends only on `cluster`, but scenario startup creates and activates the sandbox and loads the scenario without necessarily changing `cluster`. That means a scenario can become active while `currentContext` still points at the old global cluster, so commands can miss scenario faults or mutate the wrong state.  
**Recommendation:** Key this effect off `activeScenario?.id` and `cluster`, and derive the active context from reactive store state rather than `getState()` inside the effect. Effort: S

**[SEVERITY: High] [AREA: Logic]**  
**File(s):** `src/store/simulationStore.ts:476`, `src/utils/scenarioLoader.ts:307`  
**Finding:** Exiting a scenario overwrites the global simulator back to canned healthy values instead of restoring pre-scenario state.  
**Detail:** `exitScenario()` clears contexts and then calls `clearAllFaults()`, which hard-resets every GPU and node to fixed values like `temperature: 45`, `powerDraw: 300`, `utilization: 0`. That destroys legitimate non-scenario simulator state and makes sandbox teardown bleed into the global model.  
**Recommendation:** Snapshot the global cluster before scenario entry and restore that snapshot on exit. Cleanup code should never mutate the global cluster to arbitrary defaults. Effort: M

**[SEVERITY: High] [AREA: Learning/UX]**  
**File(s):** `src/components/PerformanceBenchmark.tsx:19`, `src/components/PerformanceBenchmark.tsx:119`, `src/components/PerformanceBenchmark.tsx:165`, `src/store/learningStore.ts:452`  
**Finding:** The readiness benchmark widget produces mathematically bogus pass predictions and percentiles.  
**Detail:** `getReadinessScore()` already returns `0-100`, but `PerformanceBenchmark` multiplies it by `100` again for percentile and pass prediction. So a readiness of `65` becomes `6500%`, which effectively saturates the percentile and "high likelihood of passing" logic. It also memoizes `overallPercentile` off the stable action reference instead of the numeric score, and the file openly states the cohort data is simulated.  
**Recommendation:** Remove the extra `* 100`, memoize on a real numeric dependency, and either hide percentile/pass prediction until you have real cohort data or label it as synthetic. Effort: S

**[SEVERITY: High] [AREA: Learning]**  
**File(s):** `src/components/ExamWorkspace.tsx:154`, `src/store/learningStore.ts:385`, `src/components/PerformanceBenchmark.tsx:82`  
**Finding:** Full practice exams never reach the `examAttempts` history used by analytics.  
**Detail:** The store has an `addExamAttempt()` action, and analytics components read `examAttempts`, but the exam submit path never calls that action. So even after you fix scoring, the benchmark/trend screens still will not reflect actual full-exam performance.  
**Recommendation:** Append the real breakdown to `examAttempts` from the submit and timeout paths before leaving the workspace. Add a regression test that checks the history length after an exam. Effort: S

**[SEVERITY: High] [AREA: Learning/Arch]**  
**File(s):** `src/data/narrativeScenarios.json:15`, `src/data/narrativeScenarios.json:195`, `src/components/LabWorkspace.tsx:142`, `src/components/ExamGauntlet.tsx:105`  
**Finding:** The advertised 3-tier progression is mostly metadata theater.  
**Detail:** Only two scenarios in `narrativeScenarios.json` actually declare a `tier`; the other 30 omit it. `LabWorkspace` treats missing tier as effectively always unlocked, and `ExamGauntlet` forcibly assigns every scenario to tier 2 anyway. The progression system cannot be meaningful if the content does not carry the tier data.  
**Recommendation:** Add explicit tier metadata to every scenario and enforce it consistently, or remove tier language from the product until the content is actually tiered. Effort: L

**[SEVERITY: High] [AREA: Architecture/Logic]**  
**File(s):** `src/hooks/useCloudSync.ts:27`, `src/hooks/useCloudSync.ts:147`, `src/hooks/useCloudSync.ts:180`, `src/hooks/useCloudSync.ts:227`, `src/hooks/useMetricsSimulation.ts:78`  
**Finding:** Cloud sync has race conditions, incomplete restore behavior, and can defer sync indefinitely during active simulation.  
**Detail:** Login triggers `pullAndMerge()` twice: once from the auth event and again from the "logged in" subscription effect. The merged simulation restore only imports `cluster`, silently dropping `systemType`, `simulationSpeed`, `scenarioProgress`, and `completedScenarios`. Also, while the simulator is running, the simulation store mutates every second, so the 5-second debounce keeps getting reset and may never flush until activity stops.  
**Recommendation:** Do one authenticated pull per session, apply all merged sim fields, suppress outgoing sync while merging, and checkpoint on meaningful events rather than every mutation. Effort: L

**[SEVERITY: High] [AREA: Security]**  
**File(s):** `src/utils/cloudSync.ts:49`, `src/hooks/useCloudSync.ts:153`, `src/store/simulationStore.ts:708`, `src/store/learningProgressStore.ts:538`, `src/store/learningStore.ts:502`  
**Finding:** Persisted and cloud-synced state is trusted without schema validation.  
**Detail:** Raw `JSON.parse()` output is pushed into stores and `importCluster()` with no structural validation. Both learning stores persist without `version` or `migrate`, so tampered or stale payloads can poison the app shape and only fail later in rendering or scoring code.  
**Recommendation:** Validate all persisted and cloud payloads with a runtime schema, add store versions plus migrations, and drop invalid records back to defaults with user-visible recovery messaging. Effort: M

**[SEVERITY: High] [AREA: Realism]**  
**File(s):** `src/simulators/benchmarkSimulator.ts:980`, `src/simulators/benchmarkSimulator.ts:1018`  
**Finding:** NCCL benchmark outputs are disconnected from injected GPU and fabric faults.  
**Detail:** The all-reduce bandwidth and latency calculations depend only on hardware specs, message size, and node count. XID faults, ECC issues, HCA link errors, and NVLink degradation do not feed the model at all, so the simulator teaches no causal relationship between failures and observed NCCL behavior.  
**Recommendation:** Fold GPU health, HCA error counters, link state, and topology penalties into the benchmark model so faults actually change throughput and latency. Effort: L

## Medium

**[SEVERITY: Medium] [AREA: Learning]**  
**File(s):** `src/data/examQuestions.json:14`, `src/utils/examEngine.ts:334`  
**Finding:** The 199-question bank is not actually balanced to the claimed NCP-AII domain weights.  
**Detail:** The bank breaks down as `domain1 54 (27.1%)`, `domain2 12 (6.0%)`, `domain3 33 (16.6%)`, `domain4 61 (30.7%)`, `domain5 39 (19.6%)`, versus the target `31/5/19/33/12`. Weighted exam selection masks some of this in full-practice mode, but domain tests, weak-area focus, and content authoring priorities are still skewed, especially toward troubleshooting.  
**Recommendation:** Rebalance the source bank, not just the sampler, and add a build-time content audit that fails if domain drift exceeds a threshold. Effort: M

**[SEVERITY: Medium] [AREA: Logic/Realism]**  
**File(s):** `src/components/Terminal.tsx:869`, `src/components/Terminal.tsx:1176`  
**Finding:** Bare environment assignments succeed without changing the environment.  
**Detail:** `export VAR=value` mutates `currentContext.current.environment`, but plain `VAR=value` only echoes the text and exits `0`. Any shell workflow relying on temporary env assignment will silently behave wrong.  
**Recommendation:** Parse bare assignments into the shell context, or reject them explicitly if you do not want to support them. Effort: S

**[SEVERITY: Medium] [AREA: Logic/Realism]**  
**File(s):** `src/components/Terminal.tsx:1115`, `src/components/Terminal.tsx:1211`, `src/utils/pipeHandler.ts:366`  
**Finding:** Shell substitution and pipeline behavior are permissive in ways that teach wrong habits.  
**Detail:** `$(...)` is replaced with hardcoded `12345` or `0`, and unknown pipe stages are silently passed through unchanged. That means invalid shell constructs can appear to work, and scenario validation can be influenced by fake expansion rather than real command behavior.  
**Recommendation:** Implement a narrow supported subset faithfully, and fail fast on unsupported shell features instead of pretending they worked. Effort: M

**[SEVERITY: Medium] [AREA: UX/Logic]**  
**File(s):** `src/components/ExamWorkspace.tsx:137`, `src/components/ExamWorkspace.tsx:190`, `src/components/ExamWorkspace.tsx:784`  
**Finding:** The flag-question UI is a dead control.  
**Detail:** `handleToggleFlag()` builds a new flagged list and then does nothing with it. The rest of the workspace reads `activeExam.flaggedQuestions`, so the UI offers a review workflow that never actually records flags.  
**Recommendation:** Add a store action for question flagging or remove the control until it exists. Effort: S

**[SEVERITY: Medium] [AREA: UX]**  
**File(s):** `src/components/AppHeader.tsx:176`, `src/App.tsx:325`, `src/components/SimulatorView.tsx:536`  
**Finding:** Accessibility semantics are present but not truthful.  
**Detail:** The nav uses `role="tab"` and `aria-controls="panel-*"` but the app renders a single `tabpanel` with id `main-content`, not matching controlled panel ids. The split handle is keyboard-focusable as a separator but has no keyboard resize handler or value semantics, so it advertises accessibility it does not actually provide.  
**Recommendation:** Either implement the full tab and tabpanel contract and keyboard resize behavior, or simplify to honest button semantics. Effort: M

**[SEVERITY: Medium] [AREA: UX]**  
**File(s):** `src/components/WelcomeScreen.tsx:137`, `src/components/LabsAndScenariosView.tsx:112`  
**Finding:** First-time onboarding is mostly marketing copy, not guided action.  
**Detail:** The welcome modal sells the product well, then drops the user into a flat mission grid with no recommended first mission, no "start here" path, and no explanation of the progression model. Domain 0 foundational content exists, but the UI does not strongly steer beginners into it.  
**Recommendation:** Add a single-start onboarding path, default it to the domain 0 missions, and surface the next recommended action after each completion. Effort: M

**[SEVERITY: Medium] [AREA: UX]**  
**File(s):** `src/components/AppHeader.tsx:70`, `src/components/SimulatorView.tsx:27`, `src/components/SimulatorView.tsx:470`  
**Finding:** The product's small-screen story is internally contradictory.  
**Detail:** The header warns everything below `1280px` is a compromised experience, but the simulator also has an explicit mobile breakpoint at `768px` and a separate tabbed layout. Between `768` and `1279`, the app simultaneously tries to support the viewport and tells the user not to trust it.  
**Recommendation:** Define the actually supported breakpoints, optimize the tablet band intentionally, and replace the generic warning with capability-specific fallbacks. Effort: M

**[SEVERITY: Medium] [AREA: Performance]**  
**File(s):** `src/App.tsx:111`, `src/components/SimulatorView.tsx:470`, `src/components/Terminal.tsx:140`, `vite.config.ts:27`  
**Finding:** The default route is doing too much work up front for a browser app of this size.  
**Detail:** The current build ships a `1,133.8 KB` main chunk, plus heavy vendor chunks for xterm and charts. `App` subscribes to the whole simulation store, `Terminal` eagerly constructs 20 simulator instances, and `SimulatorView` mounts `Dashboard` and `Terminal` in the `containerWidth === 0` branch and then mounts them again after measurement.  
**Recommendation:** Use selectors in `App`, lazy-load terminal and dashboard panels independently, and replace the initial double-mount branch with a cheap skeleton. Effort: M

**[SEVERITY: Medium] [AREA: Realism]**  
**File(s):** `src/components/Terminal.tsx:1059`, `src/components/Terminal.tsx:1072`, `src/components/Terminal.tsx:1103`  
**Finding:** Several exposed shell commands are no-ops or fake enough to mis-train users.  
**Detail:** `sleep` returns immediately, `date` emits the browser locale string, and `rm/cp/mv/chmod/chown` all succeed without changing any state. This gives the CLI breadth, but not operational credibility.  
**Recommendation:** Fail unsupported commands honestly, or invest in a small virtual filesystem and process model so supported commands have consequences. Effort: L

**[SEVERITY: Medium] [AREA: Realism]**  
**File(s):** `src/simulators/nvidiaSmiSimulator.ts:923`  
**Finding:** `nvidia-smi --gpu-reset` teaches a restriction that the real tool does not have.  
**Detail:** The simulator requires `-i` and errors otherwise. NVIDIA's nvidia-smi documentation documents all-GPU reset behavior when no specific ID is supplied.  
**Recommendation:** Support the all-GPU case or explicitly document the simulator limitation in help text so users are not learning a false rule. Effort: S

**[SEVERITY: Medium] [AREA: Realism]**  
**File(s):** `src/simulators/dcgmiSimulator.ts:43`, `src/simulators/dcgmiSimulator.ts:444`  
**Finding:** `dcgmi diag` is modeled as a smaller command surface than the real tool.  
**Detail:** The simulator only accepts diagnostic modes `1-3`. NVIDIA's DCGM diagnostics docs describe four run levels, with extra software and hardware coverage beyond what this implementation exposes.  
**Recommendation:** Extend the simulator to level 4 or narrow the teaching copy so it stops implying fidelity it does not have. Effort: M

**[SEVERITY: Medium] [AREA: Realism]**  
**File(s):** `src/simulators/infinibandSimulator.ts:193`  
**Finding:** `iblinkinfo` is implemented as a local-port dump, not a fabric link report.  
**Detail:** The real tool is about reporting link information across the InfiniBand fabric. NVIDIA's InfiniBand Fabric Utilities docs frame it as topology-level link inspection. This implementation only prints the current node's HCA ports, which is closer to `ibstat` than `iblinkinfo`.  
**Recommendation:** Add peer and switch endpoint reporting to `iblinkinfo`, or steer topology inspection toward `ibnetdiscover` and clearly scope `iblinkinfo` as simplified. Effort: M

**[SEVERITY: Medium] [AREA: Logic/Architecture]**  
**File(s):** `src/store/scenarioContext.ts:68`, `src/store/scenarioContext.ts:378`  
**Finding:** The scenario isolation API is brittle even before you consider the bugs around it.  
**Detail:** `getCluster()` returns the mutable internal cluster object directly, and `applyToGlobalState()` does not handle mutation types it declares elsewhere, including `node-add` and `node-update`. That makes the sandbox contract easy to violate and incomplete if merge-back ever becomes a real feature.  
**Recommendation:** Return readonly views or clones from `getCluster()`, and either implement all mutation types or narrow the public mutation API. Effort: M

## Brief Positives

Cross-tool XID propagation is better than average: `nvidia-smi`, `dmesg`, NVSM, and PCI-facing tools are at least trying to reflect shared GPU fault state. The power and temperature coupling in the metrics simulation is also more defensible than most browser-only infrastructure simulators.

## Top 5 Quick Wins

1. Persist the real exam breakdown and call `addExamAttempt()` from the submit and timeout path.
2. Remove the readiness `*100` bug and stop presenting synthetic percentile and pass prediction as real analytics.
3. Rebind terminal scenario context on `activeScenario?.id`, not just `cluster`, and replace `clearAllFaults()` teardown with snapshot restore.
4. Remove gauntlet `Mark Complete` until completion can be proven by scenario validation.
5. Add runtime schema validation plus store versioning and migrations for local and cloud state.

## Top 5 Strategic Investments

1. Rebuild the exam subsystem around authoritative attempt objects, scoring, review history, and item analytics.
2. Make tiers real: explicit tier metadata on every scenario, unlocks based on demonstrated scenario performance, and no fallback "standard" ambiguity.
3. Split `src/components/Terminal.tsx` into shell/session, builtin commands, simulator registry, output post-processing, and scenario integration layers.
4. Replace ad hoc sandbox cleanup with snapshot-based scenario isolation and explicit merge and restore semantics.
5. Add a causal realism layer for GPU and fabric faults and pair it with an after-action coach or AI tutor that explains why a command worked, failed, or was suboptimal.
